03 вҖ” Measurement Models & Mahalanobis Gating

Why OKS sometimes rejects sensorbar updates

Prerequisite: 02-kalman-filter.md (EKF predict/update equations, covariance growth) Unlocks: 04-imu-fusion.md (theta update, gyro bias), 05-failure-modes.md (diagnosing silent rejections)

Why Should I Care? (OKS Project Context)

You are staring at a bag where the robot drifts 0.3m off a lane line over 8 seconds, yet the sensorbar is firing. The estimator logs show nothing вҖ” no explicit rejection messages, no errors. The sensorbar is silently ignored. This happens every day on the SCL floor.

Understanding measurement models tells you exactly why: - Mahalanobis gating can reject a perfectly good sensorbar reading if the state covariance is too tight (P too small) вҖ” the measurement looks like an outlier even when the robot is close to the line. - OKS does not use a standard Kalman gain for XY updates. It clamps the state to a range and sets covariance to the variance of a uniform distribution. Knowing this lets you compute the exact threshold at which a sensorbar range “wins” over the current covariance. - is_reliable=True does not mean the reading is accurate. Sensorbar duplicates (11вҖ“30% of frames in SCL bags) pass the reliability gate but carry identical stale offsets. They look like valid measurements to the estimator.

By the end of this chapter you will be able to: 1. Compute whether a given sensorbar reading will be accepted or rejected without running code. 2. Explain why the clamp update is mathematically equivalent to a uniform-likelihood measurement. 3. Identify in a bag trace the difference between “measurement rejected by Mahalanobis gate” and “measurement accepted but wrong (duplicate)”.

PART 1 вҖ” WHAT IS A MEASUREMENT MODEL

1.1 The Sensor-to-State Relationship

The state vector x holds what we believe about the robot:

        в”Ң x  в”җ    вҶҗ world-frame X position (metres)
    x = в”Ӯ y  в”Ӯ    вҶҗ world-frame Y position (metres)
        в”” Оё  в”ҳ    вҶҗ heading angle (radians, CCW from +X)

The measurement z is what a sensor actually reports. A measurement model is the function that predicts what the sensor should report if the state were exactly x:

    z = h(x) + noise

For a linear relationship we write h(x) = H x where H is the measurement matrix.

Example вҖ” GPS measuring position directly:

         в”Ң 1  0  0 в”җ   в”Ң x в”җ   в”Ң x в”җ
    z = H x =       в”Ӯ   в”Ӯ y в”Ӯ = в”Ӯ y в”Ӯ
         в”” 0  1  0 в”ҳ   в”” Оё в”ҳ   в”” y в”ҳ

    H = [1 0 0]   (measuring x only)
    H = [0 1 0]   (measuring y only)

GPS gives us two independent scalar measurements, each with its own H row.

Key insight: H tells the filter which component of the state the sensor is observing. A row of H like [0, 1, 0] means “the sensor only sees y; x and Оё are invisible to it.”

When the robot passes over a floor line, the detectors that cross the line return a different reflectance signal. By finding the centroid of activated sensors, the firmware computes the lateral offset of the robot from the line.

2.2 The Measurement Range (Not a Point!)

Unlike a GPS fix that gives a precise coordinate, the sensorbar gives a range:

    Sensorbar has N sensors, each 6mm apart.
    If 3 sensors are activated:

    в”Җв”Җв”Җ Sensor 1 (6mm spacing) в”Җв”Җв”Җ Sensor 2 в”Җв”Җв”Җ Sensor 3 в”Җв”Җв”Җ

    The line is somewhere within the activated span.
    Centroid computation gives Вұ3mm resolution.

    Typical reported range:    [meas.min, meas.max]
                                e.g. [1.985m, 2.015m]
                                    вҶ‘ 3cm total width вҶ‘

This is why OKS uses a clamping update rather than a Kalman gain вҖ” the measurement is honestly a range, not a point. The uniform distribution is the right model for “the robot is somewhere in this 3cm band, with equal probability anywhere in the band.”

PART 3 вҖ” INNOVATION AND MAHALANOBIS DISTANCE

3.1 What Is Innovation?

Before the filter accepts a measurement, it checks consistency. The innovation is:

    ОҪ = z - h(xМӮвҒ»)

    where:
      z      = actual measurement from sensor
      xМӮвҒ»    = prior state estimate (before this measurement)
      h(xМӮвҒ») = predicted measurement (what sensor SHOULD read if xМӮвҒ» is correct)

For the linear case: ОҪ = z - H xМӮвҒ»

  • Small ОҪ вҶ’ measurement consistent with prediction вҶ’ probably accept
  • Large ОҪ вҶ’ measurement far from prediction вҶ’ possibly an outlier or sensor fault

But what is “large”? 0.1m is large if the state uncertainty is 0.001m, but tiny if uncertainty is 1.0m. The filter needs to normalize by uncertainty.

3.2 Mahalanobis Distance

The Mahalanobis distance is the innovation normalized by the total uncertainty in the measurement space:

                                              вҒ»В№
    dВІ = ОҪбөҖ  (H PвҒ» HбөҖ + R)   ОҪ

    where:
      PвҒ»  = prior state covariance (uncertainty before this measurement)
      R   = measurement noise covariance (sensor uncertainty)
      H   = measurement matrix

    S = H PвҒ» HбөҖ + R    вҶҗ "innovation covariance"
                           (total uncertainty in measurement space)

    dВІ = ОҪбөҖ SвҒ»В№ ОҪ

Scalar case (one measurement, one uncertain state dimension):

    d = |z - predicted| / sqrt(P_component + R)

P_component + R is the sum of “how uncertain we are about the state” plus “how uncertain the sensor is.” Together they define the expected spread of the innovation.

3.4 Worked Example вҖ” Mahalanobis Distance

Scenario: Robot moving east, approaches a N-S line at X = 2.00m.

Given:

    State estimate:      xМӮвҒ»(x) = 2.05m   (dead-reckoning puts us 5cm east of line)
    State covariance:    PвҒ»(x,x) = 0.010 mВІ
    Measurement:         z = 2.00m         (sensorbar reports line at exactly this X)
    Measurement noise:   R = 0.005 mВІ      (sensorbar has ~7cm std dev)

Step 1: Compute innovation

    ОҪ = z - H xМӮвҒ» = 2.00 - 2.05 = -0.05m

Step 2: Compute innovation standard deviation

    S = H PвҒ» HбөҖ + R
      = PвҒ»(x,x) + R          (H = [1,0,0] so H PвҒ» HбөҖ = PвҒ»(x,x))
      = 0.010 + 0.005
      = 0.015 mВІ

    sqrt(S) = 0.122m

Step 3: Mahalanobis distance

    d = |ОҪ| / sqrt(S) = 0.05 / 0.122 = 0.41

Decision: ACCEPTED (d = 0.41 < 2.0) вң“

Now try with very tight covariance:

    PвҒ»(x,x) = 0.001 mВІ   (state is very confident вҖ” tight covariance)

    S = 0.001 + 0.005 = 0.006 mВІ
    d = 0.05 / sqrt(0.006) = 0.05 / 0.077 = 0.65

Decision: ACCEPTED (d = 0.65 < 2.0) вң“ вҖ” still passes because 5cm is within 2Пғ.

Now try with a large drift:

    Innovation = 0.4m  (robot is 40cm from the line вҖ” something went wrong)
    PвҒ»(x,x) = 0.010 mВІ

    S = 0.015 mВІ
    d = 0.4 / 0.122 = 3.28

Decision: REJECTED (d = 3.28 > 2.0) вң— вҖ” the measurement is 3.3Пғ away from prediction.

This is the “silent rejection” scenario. The sensorbar fired, the measurement was valid, but the filter rejected it because dead-reckoning had drifted so far that the correct position appeared to be an outlier.

3.5 The Danger Zone: When Tight P Causes Rejections

Counter-intuitive: A tighter covariance P can cause MORE rejections, not fewer.

    Scenario: Robot has been on a long straight with many sensorbar updates.
    PвҒ»(x,x) is driven very low, say 0.0001 mВІ.

    Next measurement has a genuine 8cm offset (real drift, not outlier):
    S = 0.0001 + 0.005 = 0.0051 mВІ
    d = 0.08 / sqrt(0.0051) = 0.08 / 0.071 = 1.12    вҶҗ still accepted

    But at 12cm offset:
    d = 0.12 / 0.071 = 1.68    вҶҗ still accepted

    At 15cm offset:
    d = 0.15 / 0.071 = 2.11    вҶҗ REJECTED

    With looser covariance P = 0.010:
    d = 0.15 / 0.122 = 1.23    вҶҗ accepted!

Key insight: After a slip event, P gets inflated to infinity (estimator detects slip and sets P вҶ’ вҲһ). This WIDENS the acceptance gate, allowing the next sensorbar update to be accepted even with a large innovation вҖ” which is exactly correct behavior. The filter recovers by accepting a measurement it would have rejected pre-slip.

PART 4 вҖ” OKS CLAMP UPDATE (NON-STANDARD KALMAN)

4.1 Standard Kalman Update vs OKS Clamp Update

Standard Kalman filter computes a gain K and updates the state:

    Standard KF update:
    K = PвҒ» HбөҖ (H PвҒ» HбөҖ + R)вҒ»В№       вҶҗ Kalman gain
    xМӮвҒә = xМӮвҒ» + K (z - H xМӮвҒ»)          вҶҗ state update (weighted blend)
    PвҒә = (I - K H) PвҒ»                 вҶҗ covariance update

    K is between 0 and 1:
      K вҶ’ 0: sensor has much higher noise than state uncertainty вҶ’ barely moves
      K вҶ’ 1: sensor is much more accurate than state вҶ’ jump to sensor reading

OKS XY update (from ``):

    OKS clamp update:
    xМӮвҒә(x) = clamp(xМӮвҒ»(x), meas.x_min, meas.x_max)
    PвҒә(x,x) = min(PвҒ»(x,x), (meas.x_range)ВІ / 12)

    where meas.x_range = meas.x_max - meas.x_min

These are NOT the same operation. Why does OKS use clamp instead of Kalman gain?

The clamp operation is geometrically equivalent to multiplying the Gaussian state by the uniform measurement likelihood and renormalizing вҖ” the result is the state truncated to the measurement range.

4.3 The Variance-of-Uniform Formula

For a uniform distribution over [a, b]:

         (b - a)ВІ
    ПғВІ = в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җ
             12

    Derivation:
      E[X] = (a + b) / 2
      Var[X] = E[(X - Ој)ВІ]
             = вҲ«_a^b (x - (a+b)/2)ВІ В· (1/(b-a)) dx
             = (b-a)ВІ / 12    вң“

OKS sensorbar range example:

    Range = 0.06m (6cm span, Вұ3cm from line)
    a = 1.970m, b = 2.030m

    ПғВІ_uniform = (0.06)ВІ / 12 = 0.0036 / 12 = 0.0003 mВІ
    Пғ_uniform  = 0.0173m = 1.73cm

The covariance update rule PвҒә = min(PвҒ», ПғВІ_uniform) says: - If the state was already more certain than the sensorbar вҶ’ keep the current covariance. - If the state was less certain (larger P) вҶ’ tighten it to the sensorbar’s uniform variance.

A single sensorbar crossing is ambiguous between: - The robot is near the intended line at the predicted position, or - The robot is near the adjacent line (robot has drifted by Вұ1 line_spacing)

5.2 Probability Ratio Gate

OKS resolves the ambiguity with a likelihood ratio test:

    Hypothesis H0: robot is at the primary line (consistent with state estimate)
    Hypothesis H1: robot is at the secondary line (N lines away)

    Compute p(z | H0) and p(z | H1) using the current state distribution.

    If p(z | H1) / p(z | H0) > threshold:
        вҶ’ ambiguous вҶ’ REJECT measurement
    Else:
        вҶ’ H0 is sufficiently more likely вҶ’ ACCEPT

In practice, if the robot state estimate is well within a lane (P is small), H1 requires the robot to be ~1.5m away from its estimated position вҖ” extremely unlikely вҖ” so the ratio is tiny and the measurement is accepted. But if P is large (after a slip, or after long dead-reckoning), H1 becomes plausible, and the measurement may be rejected.

6.2 The is_reliable Trap вҖ” Why Duplicates Are Invisible

    Timeline of sensorbar messages at 50Hz (20ms intervals):

    t=0ms    msg: {offset=-0.015m, is_reliable=True}   вҶҗ fresh reading
    t=20ms   msg: {offset=-0.015m, is_reliable=True}   вҶҗ DUPLICATE (SPI retry)
    t=40ms   msg: {offset=-0.015m, is_reliable=True}   вҶҗ DUPLICATE
    t=60ms   msg: {offset=-0.018m, is_reliable=True}   вҶҗ fresh reading (robot moved)

All four messages have is_reliable=True. The estimator’s is_reliable gate at only catches SPI frame misses and all-zero readings (hardware-level failures). It has no concept of “stale data from a previous read cycle.”

The filter’s perspective: - It sees three measurements at t=0ms, t=20ms, t=40ms that all say -0.015m - Each one passes the Mahalanobis gate (innovation stays near zero since state isn’t changing between them вҖ” or covariance is tight from the first update) - Each one executes the clamp update, re-confirming the same position - P is driven even lower each time (the filter becomes MORE confident) - When the fresh reading at t=60ms arrives, P is very tight - If the robot has actually moved 5cm and the fresh reading reflects this, the innovation may now EXCEED the 2Пғ gate because P is so tight

    Result: Duplicates вҶ’ over-confident state вҶ’ genuine updates rejected

    This is the core mechanism behind "silent rejection" in SCL bags.

6.3 Theta Update Is Different (Standard Kalman)

For heading Оё, OKS uses the standard Kalman update with gyro bias correction:

    z_Оё = measured heading from sensorbar (line direction gives Оё constraint)
    bias = estimated gyro bias (states b_x, b_y, b_z)

    H_Оё = [0, 0, 1, 0, 0, 0, ...]   вҶҗ only sees Оё, not XY

    K = PвҒ» HбөҖ / (H PвҒ» HбөҖ + R_Оё)   вҶҗ standard Kalman gain
    ОёМӮвҒә = ОёМӮвҒ» + K (z_Оё - ОёМӮвҒ»)        вҶҗ weighted update
    PвҒә = (1 - K H) PвҒ»               вҶҗ covariance shrinks

The reason theta uses a proper Kalman gain while XY uses clamp: - Оё measurement from the sensorbar line direction IS a precise point measurement (the line has a known orientation, and the robot either aligns or doesn’t) вҖ” Gaussian noise is honest. - XY measurement is a range (the robot is somewhere within the IR span) вҖ” uniform is honest.
